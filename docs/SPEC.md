# SPEC.md — 产品规格与验收标准

> 本文档定义购物推荐 Agent 的产品功能规格和验收标准。所有定义基于产品设计文档 (`architecture_consolidated.md`)、工程分析 (`engineering_analysis.md`) 和交接审阅文档 (`review_findings.md`) 中已确认的设计决策。

---

## 一、产品概述

### 1.1 产品定位

通用购物决策 Agent——面向各种**存在客观可比参数、决策维度多、不同使用场景下选择差异大**的品类。

**目标**：通过多轮对话帮助用户理解自己的真实需求、获取结构化的品类知识和产品信息，最终降低购买决策成本。产品的目的是帮用户减小决策成本，不是替用户做决定。

产品既支持典型的**单品选购**，也支持围绕当前购物目标展开的**搭配/补齐/升级**类决策；长期目标是通过跨会话记忆和跨品类迁移，逐步形成「越用越懂」的购物决策体验。

### 1.2 适用品类

**适用品类特征：**
- 存在一定量的客观可量化参数（性能、规格、测试数据等）
- 决策维度多且相互制约（存在 tradeoff）
- 不同使用场景对产品需求差异显著
- 用户可能不了解技术细节，也不一定清楚自己的真实需求

**典型适用品类示例**：户外装备、手机/数码、相机/镜头、汽车、家电、智能家居、办公设备、运动器材、乐器等。

**不适用的品类**：纯粹由审美/个人风格驱动的品类（如服饰搭配、艺术品）。灰色地带品类（耳机、手表、背包等——既有功能性又有审美/社会属性）可以通过 JTBD 框架部分覆盖。

> [!NOTE]
> 本系统不限制品类范围。文档中使用的示例（如户外装备、智能家居、摄影器材等）仅为说明性案例，不代表系统局限于这些品类。系统设计为通用框架，适用于任何符合上述特征的消费品类。

### 1.3 V1 范围边界

| 范围内 | 范围外 |
|-------|-------|
| 单用户本地使用 | 多用户支持 |
| CLI 交互界面 | Web/移动端 UI |
| 串行执行（所有操作按顺序进行） | 并行执行 |
| 本地 JSON 文件持久化 | 数据库存储 |
| 1+1 Agent 架构（主 Agent + 研究 Sub-Agent） | 多角色 Agent 接力链 |

### 1.4 支持的任务形态

V1 的主任务仍以当前购物需求为中心，但不限制为”只能选一个单品”。典型任务包括：

- **单品选购**：如「想买一件冲锋衣」「想买个智能音箱」「想买台笔记本电脑」
- **搭配/补齐**：如「帮我补齐一套周末徒步装备」「为新家配置智能家居系统」「配齐一套摄影器材」
- **升级/换代**：如「之前这双登山鞋不太行，想升级」「手机用了三年想换新的」「相机镜头想升级」
- **跨品类延伸**：如「已经买了冲锋衣，接下来还需要什么」「有了相机机身，还需要配什么镜头」

这些任务共享同一套对话式需求挖掘和研究流程。系统不要求用户预先维护完整的已有物品清单，只在当前任务确实需要时按需追问最相关的已有物品或缺失项。

> [!NOTE]
> 以上示例涵盖多个品类（户外装备、智能家居、数码产品、摄影器材等），仅为说明任务形态的多样性。系统不限制品类范围。

> [!NOTE]
> V1 对搭配/补齐/升级类任务采用**单焦点顺序推进**：系统允许围绕同一购物目标连续处理多个子问题，但在任一时刻只围绕一个当前 `product_type` 推进，不承诺同时保留多个产品类型的并行候选池，也不以内建状态模型支持跨焦点统一排序。

## 二、核心架构概述

> 详细架构设计见 ARCHITECTURE.md。此处仅概述产品层面的架构选型。

### 2.1 1+1 混合架构

系统由两个 Agent 组成：

| Agent | 职责 | 模型级别 |
|-------|------|---------|
| **主 Agent（Orchestrator）** | 需求挖掘、流程决策、对比分析与推荐、维护全局状态 | 由 env 指定的强推理模型 |
| **研究 Sub-Agent（Research Agent）** | 执行互联网信息搜索、提取和结构化产品信息、初步质量判断 | 由 env 指定的低成本高能力模型 |

**主 Agent 是纯推理器**——不拥有任何工具（FunctionTool），每次调用接收 context、做一次推理、输出结构化决策。

**研究 Sub-Agent 有搜索工具**——每次任务创建新实例，拥有独立的 context window，任务完成后销毁。

### 2.2 四层文档化记忆体系

| 文档层 | 内容 | 生命周期 | 更新频率 |
|-------|------|---------|---------|
| **品类领域知识**（`knowledge/`） | 品类决策维度、tradeoff、场景映射、常见误区、品牌全景 | 长期沉淀 | 研究 Agent 完成品类调研时 |
| **宏观画像**（`user_profile/global_profile.json`） | 跨品类稳定特征、消费风格、生活方式标签 | 跨会话持久 | 推荐完成后生成草稿，下次启动恢复检查后应用 |
| **品类偏好**（`user_profile/category_preferences/`） | 主要使用场景、品牌偏好、品类消费倾向、选购记录 | 跨会话持久 | 推荐完成后生成草稿，下次启动恢复检查后应用 |
| **会话状态**（`sessions/`） | 决策推进状态 + 需求画像快照 + 错误追踪 | 会话级 | 每轮对话后即时更新 |

## 三、用户交互流程

### 3.1 六阶段流程概览

整个购物推荐会话可划分为六个阶段。这些阶段**不是严格的线性流水线**——后续阶段中可能发现信息不足而回退到前序阶段，但整体推进方向是从左到右。

对于搭配/补齐/升级类任务，这六个阶段仍然成立，但其含义会更宽一些：主 Agent 可以围绕同一个购物目标，按顺序分多次需求澄清、分多次搜索、再做阶段性分析，而不要求所有任务都被压缩成单次单品闭环。

```
 0.初始化  →  1.品类认知构建  →  2.需求挖掘  ⇄  3.产品搜索  →  4.两轮推荐  →  5.会话收尾
```

| 阶段 | 执行者 | 核心任务 | 进入条件 | 退出条件 |
|------|--------|---------|---------|---------|
| **0. 初始化** | 系统 + 主 Agent | 检测新用户并执行 onboarding（性别/年龄段/城市）；加载已有文档 | 会话开始 | 文档加载完成 |
| **1. 品类认知构建** | 研究 Sub-Agent | 调研品类决策维度、tradeoff、场景映射、常见误区 | knowledge 中无对应产品类型 section | 品类知识文档填充完成 |
| **2. 需求挖掘** | 主 Agent | 批量提问核心维度；分析响应后检查 urgency 阈值；仅针对阻塞维度追问；必要时确认已有物品/缺失项 | 品类知识已加载 | priority ≥ 3 的维度中无 urgency ≥ 12 的阻塞项，且 urgency 8-11 区间维度不超过 3 个 |
| **3. 产品搜索** | 研究 Sub-Agent | 搜索具体产品评测、参数、评价；初步匹配筛选。复杂任务下可由主 Agent 分多次调度 | 核心需求已明确 | 返回结构化产品列表 |
| **4. 两轮推荐** | 主 Agent | 默认按“第一轮广度推荐 → 收集反馈 → 第二轮深度分析”推进；必要时在候选分析与补充搜索之间往返 | 收到产品数据 | 用户获得足够决策支持 |
| **5. 会话收尾** | 系统自动 | 生成并保存 `pending_profile_updates`，保留当前 session 供下次恢复检查 | 推荐周期完成，或用户在当前轮获得足够决策支持 | 画像草稿已保存，session 可恢复 |

### 3.2 阶段间回退关系

- **3 → 2 回退**：产品搜索结果暴露了之前需求挖掘的信息不足（如搜出的产品在某个未讨论的维度上差异巨大）→ 回到需求挖掘确认该维度
- **4 → 2 回退**：用户在推荐阶段的反馈暴露了新的需求或偏好变化 → 更新需求画像（可能触发重新搜索）
- **4 → 3 回退**：用户提出明确新需求导致现有候选不再适用 → 带新条件重新搜索

### 3.3 Onboarding（首次用户）

**触发条件**：`global_profile.json` 不存在或 demographics 不完整（确定性系统行为）。

**执行流程**：
1. 系统在 context 中注入标注，指引主 Agent 执行 onboarding
2. 主 Agent 向用户提问性别、年龄段、所在城市（2-3 个问题，10 秒完成）
   - 进入 onboarding 模式后，当前轮只处理 demographics 收集
   - 如用户同时给出了购买诉求，先完成 onboarding，再继续正常需求对话
3. 用户回答后，主 Agent 输出 `onboard_user` action，携带 demographics 数据
4. 系统将 demographics 写入 `global_profile.json`

**设计理由**：性别、年龄段、城市这三个信息在多数品类中有功能性影响（如性别决定服装产品线），且不敏感、复用价值高。

### 3.4 搭配/补齐类任务的处理原则

当用户表达的不是单一产品需求，而是「搭配一套」「补齐还缺什么」「围绕现有装备升级」这类目标时，主 Agent 应：

1. 先把任务理解为当前购物目标，而不是强行压缩成单个 `product_type`
2. 在需求挖掘中按需询问最相关的已有物品、缺失项和预算边界
3. 按需要将问题拆成多个**顺序处理**的产品搜索子问题
4. 最终给出分步建议、阶段性优先级或当前焦点下的取舍解释；V1 不要求系统同时保留多个子问题的候选做统一横向排序

系统不要求用户预先录入完整 inventory，也不要求每次都维护长期购物计划。

## 四、需求挖掘规格

### 4.1 urgency 评分体系

每个决策维度包含三个属性：

| 属性 | 取值范围 | 含义 |
|------|---------|------|
| **priority**（权重） | 1-4 | 该维度对用户有多重要。4=关键/硬约束，3=重要，2=一般，1=无关 |
| **confidence**（置信度） | 1-4 | Agent 对该信息的确定程度。1=确定，2=模糊，3=推断，4=未知 |
| **urgency**（紧迫度） | 1-16 | `priority × confidence`，分数越高越需要优先处理 |

#### priority 详细定义

| 值 | 含义 | 对推荐的影响 |
|----|------|------------|
| 4 | 关键 — 不满足直接排除 | 硬筛选条件（硬约束） |
| 3 | 重要 — 在意但可妥协 | 高排序权重 |
| 2 | 一般 — 有影响但不决定性 | 低排序权重 |
| 1 | 无关 — 对用户没意义 | 不纳入评估 |

#### confidence 详细定义

| 值 | 含义 | 判定标准 |
|----|------|---------|
| 1 | 确定 — 可直接用于决策 | 用户直接表达 + 具体化（数字、具体型号、明确场景） |
| 2 | 模糊 — 需找机会锚定确认 | 用户表达了但使用了抽象/模糊概念 |
| 3 | 推断 — 仅作假设 | Agent 从其他信息推断，用户未直接表达 |
| 4 | 未知 — 核心维度需提问 | 完全未涉及 |

### 4.2 阈值规则

**前置条件**：仅对 **priority ≥ 3** 的维度进行阈值检查。priority ≤ 2 且 confidence=4（完全未知）属于正常状态（因为不重要所以未讨论），不应触发追问。

**系统阈值**（仅适用于 priority ≥ 3 的维度）：

| 阈值 | 含义 | Agent 行动 |
|------|------|-----------|
| urgency ≥ 12 | 存在高权重 × 高不确定的维度 | **阻塞**——不能进入推荐阶段，必须先提问/确认 |
| urgency 8-11 | 有需要关注但非阻塞的维度 | 可开始初步搜索，推荐时标注不确定性 |
| urgency ≤ 7 | 信息已充足 | 可进入产品搜索阶段 |

**中等不确定性累积规则**：即使没有 urgency ≥ 12 的阻塞维度，如果 urgency 8-11 区间的维度**超过 3 个**，搜索条件的整体不确定性过高，应先做一轮针对性追问以减少不确定维度数量至 3 个以内。

### 4.3 提问行为规范

Agent 的提问行为应遵循以下模式：

1. **初期批量提问**：基于品类知识和用户已有信息，一次性提出覆盖核心维度的问题（使用场景、预算、关键偏好等），不需要一个个单独问
2. **分析响应，批量更新**：用户的一段回答通常会同时覆盖多个维度的信息，Agent 解析后一次性更新多个维度的 priority 和 confidence
3. **阈值检查**：更新后检查 urgency 评分——是否还有 ≥12 的阻塞维度？
   - 有 → 针对性地追问/澄清这些具体维度（不是从头再遍历一遍）
   - 没有 → 直接进入产品搜索
4. **非阻塞维度不追问**：urgency 8-11 的维度可以在搜索后通过产品差异来间接澄清；urgency ≤7 的维度完全不用管。**例外**：如果 priority ≥ 3 且 urgency 8-11 区间的维度超过 3 个，整体不确定性过高，应先做一轮针对性追问
5. **已有物品按需询问**：只有当当前任务是搭配/补齐/升级，且已有物品会实质影响判断时，才追问最相关的已有物品或缺失项；不要求用户预先提供完整清单

> [!WARNING]
> **避免变成 workflow**：需求挖掘阶段不是一个逐维度遍历的清单流程。Agent 应展现自主决策能力——根据用户给出的信息量和质量，动态判断是继续追问还是直接行动。

#### priority 的三个来源（动态更新）

1. **场景推断**：品类知识文档中的 `scenario_mapping` 提供初始值
2. **用户显式表达**：用户直接说了某维度重要/不重要
3. **行为信号**：用户对产品的反馈暗示了维度偏好

### 4.4 歧义检测：锚定确认策略

**核心思路**：永远不要用抽象概念做决策，用具体数字/场景来锚定。

当用户使用以下类型的描述时，用具体化方式确认：
- **程度词**：「高海拔」「重度使用」「经常」「专业级」
- **主观评价**：「比较冷」「挺远的」
- **品类混淆**：用户可能混淆相近但不同的产品类型

品类知识文档中的 `ambiguity_risk` 字段提供品类特定的歧义点。

**示例**：
- ❌ 「好的，你需要适合高海拔的冲锋衣」
- ✅ 「你说的高海拔大概是什么范围？比如 3000-4000 米的景区步道，还是 5000 米以上需要冰雪攀登的？这两种场景需要的装备差别很大。」

### 4.5 偏好漂移检测

**两种漂移**：

| 类型 | 触发条件 | 处理方式 |
|------|---------|---------|
| **显式漂移**（用户主动说） | 用户明确改变了之前声明的偏好 | 直接更新 session 需求画像 |
| **隐式漂移**（行为暗示） | 用户的选择与之前声明不一致 | 进行偏好确认 |

**级联更新**：偏好变化 → 更新 session 需求画像 → 重新评估候选产品 → 必要时补充搜索

**异常检测**：维度权重被用户修改超过 3 次时，主动做一次需求总结确认。如持续发生，可能是纯咨询 intent，切换到低决策压力模式。

### 4.6 JTBD 需求挖掘

**核心原则**：不直接问心理需求，通过侧面问法和行为观察间接推断。JTBD 三层任务（功能/情绪/社会）作为 Agent 的思维框架，session 文档中只记录已观察到的信号。

**识别渠道**：

| 渠道 | 方式 | 适用时机 |
|------|------|---------|
| **决策旅程分享** | 提问「你之前买这类产品最看重哪些方面？」 | 首次交互或进入新品类时 |
| **关注点间接识别** | 观察用户反复追问的内容方向 | 当次对话中 |

**行为信号示例**：

| 用户行为 | 推断 |
|---------|------|
| 反复追问参数细节 | 功能驱动 |
| 关注「别人怎么评价」「主流选择」 | 社会驱动 |
| 关注「会不会后悔」「省不省心」 | 情绪驱动（风险规避） |
| 关注「送人合不合适」「拿得出手吗」 | 社会驱动（关系场景） |

**使用原则**：推断结果为内部框架，不对外呈现。

| 内部推断 | ❌ 不恰当的确认方式 | ✅ 恰当的外部表达 |
|---------|-------------------|-----------------|
| 社会需求：想通过产品表达身份 | 「你是不是想通过这个表达品位？」 | 「这款在商务场合会显得很得体」 |
| 情绪需求：怕买错后悔 | 「你是不是很怕做错决定？」 | 「这款是经过市场验证的稳妥选择」 |

### 4.7 搭配/升级类任务的轻量上下文

对于搭配、补齐、升级这类任务，Agent 可以在 session 中临时维护少量与当前目标直接相关的信息，例如：

- 当前购物目标的自然语言摘要
- 已有的关键物品
- 明确缺失的关键物品

这些信息只服务于当前会话推进，不等价于完整的长期 inventory 系统。V1 中它们的作用是帮助主 Agent 围绕同一购物目标**顺序切换当前焦点**，而不是支撑多个 product_type 候选池并行持久化。

## 五、品类知识体系规格

### 5.1 文档组织原则

- 品类知识文档（`knowledge/`）和品类偏好（`category_preferences/`）按相同的大品类维度组织，一一对应
- 品类知识文档内部分两层：**品类级通用信息**（每次加载）和**产品类型级细分信息**（按需加载）
- 文档中**不预先列出未研究的产品类型占位符**，每个产品类型的 section 只在首次产生实际需求时由研究 Agent 创建并写入

### 5.2 品类知识文档结构（两层）

| 层级 | 包含内容 | 加载时机 | 复用价值 |
|------|---------|---------|---------|
| `category_knowledge`（品类级） | 产品类型概览、跨产品类型的通用概念（如 GORE-TEX 面料、碳纤维材料）、品牌全景 | **每次涉及该品类时始终加载** | 高——如咨询冲锋衣时积累的 GORE-TEX 知识，咨询登山裤时直接复用 |
| `product_types.*`（产品类型级） | 该产品类型特有的决策维度、tradeoff、价位段、场景映射、常见误区 | **仅加载当前相关的 section** | 低——冲锋衣的 tradeoff 和登山鞋的 tradeoff 完全不同 |

**选择性加载的意义**：品类文档会随产品类型的积累不断增长。每次只加载「品类级 + 当前产品类型」可以有效控制 context 开销。

> 完整的 JSON Schema 骨架定义见 INTERFACES.md。

### 5.3 品类知识文档的生成与更新

| 场景 | 系统行为 |
|------|---------|
| 大品类文档**不存在** | 研究 Agent 从零调研，创建文档并填充 `category_knowledge` + 当前产品类型的 section |
| 大品类文档**存在**，当前产品类型 section **存在** | 加载品类级 + 当前产品类型 section |
| 大品类文档**存在**，当前产品类型 section **不存在** | 先凭已加载的 `category_knowledge` 对话，同时研究 Agent 补充当前产品类型 section |

### 5.4 `scenario_mapping` 使用机制

1. 研究 Agent 调研某产品类型后，归纳出该类型的典型使用场景大类
2. 每个场景大类映射到不同的维度优先级组合（哪些维度关键、哪些可妥协）
3. 需求挖掘阶段，用户描述自己的具体使用情况，Agent 将其映射到最接近的场景类别（或组合）
4. 以此作为各决策维度 priority 的**初始值**，后续通过对话持续修正

此映射不是精确分类，而是帮助 Agent 在对话早期快速建立维度权重的**初始化参考**。

## 六、研究 Sub-Agent 规格

### 6.1 独立性保障

- **Context 隔离**：每次研究任务创建新实例，拥有独立 context window，任务完成后销毁
- **不传回原始网页内容**，只传回结构化数据和来源链接
- 使用低成本模型，侧重搜索广度

### 6.2 两套 Prompt 模板

研究 Agent 有两种本质不同的任务类型，各自使用独立的 prompt 模板：

#### 模板一：品类调研（Category Research）

| 项目 | 规格 |
|------|------|
| **使用时机** | 用户提出新品类需求时，构建品类认知 |
| **目标** | 理解品类结构——有哪些决策维度、哪些 tradeoff、哪些场景、常见误区 |
| **输入** | 品类名称 + 产品类型 + 用户初步描述 + 搜索策略提示（可选） |
| **输出** | 品类知识文档，按通用 Schema 骨架填充（Pydantic `CategoryResearchOutput`） |
| **搜索策略侧重** | 选购指南、品类概述、专业分类文章等权威教育性内容 |
| **来源优先级** | 专业评测网站的指南 > 论坛精华帖 > 品牌官方说明 |

#### 模板二：产品搜索（Product Search）

| 项目 | 规格 |
|------|------|
| **使用时机** | 需求已基本明确，搜索具体匹配产品 |
| **目标** | 搜集具体产品的评测数据、参数、价格、用户评价，基于已有需求信息做初步匹配筛选 |
| **输入** | 明确的搜索条件（价位、场景、关键需求、排斥项）+ 搜索目标描述 + 搜索策略提示（可选） |
| **输出** | 结构化产品列表，已做初步匹配筛选，包含完整的产品功能/特点介绍（Pydantic `ProductSearchOutput`） |
| **搜索策略侧重** | 具体产品的评测文章、对比测评、用户评价、商品详情 |
| **来源优先级** | 专业评测 > 用户真实评价 > 商品详情页 > 营销内容 |

### 6.3 输出要求

`dispatch_product_search` 的 payload 至少应包含以下字段：

- `product_type`
- `search_goal`
- `constraints.budget`（可选；若当前任务预算未知、非阻塞，或用户明确要求先不看预算，调用时应显式传 `null` 或 `unspecified`）
- `constraints.key_requirements`
- `constraints.exclusions`

`ProductSearchOutput.products[*]` 中每个产品至少应包含以下字段：

- `name`
- `brand`
- `price`
- `specs`
- `features`
- `pros`
- `cons`
- `sources`
- `source_consistency`

- **`features` 字段必须包含产品功能/特点的完整介绍**，确保主 Agent 在后续对话中不需要为了回答用户关于产品具体功能的问题而重新发起搜索
- **`source_consistency` 字段**标记来源一致性（high / medium / low），供主 Agent 做语义校验
- **`search_meta` 字段**必须以结构化方式描述搜索执行状态，至少包含重试次数、结果状态、是否扩大搜索范围，供应用层稳定记录 `error_state` 和评估指标；其中 `retry_count` 和 `search_expanded` 以应用层规范化后的值为准
- 如果没有匹配的产品，**返回空列表并解释原因，不要编造产品**

### 6.4 搜索策略决策

搜索语言、关键词策略等由主 Agent 根据以下因素综合判断：
- 用户所在地区（location）
- 用户使用的语言
- 品类特征（国际化程度、地区差异）
- 具体需求（如是否关注二手市场、是否需要对比国际/国内型号）

主 Agent 在 dispatch 时通过 `research_brief` 字段（自然语言）传递搜索策略提示。`research_brief` 负责表达高层语义偏好，如搜索语言、来源优先级、特殊关注点；系统层仍负责 Tavily 工具装配、预算上限、失败重试和降级等薄护栏。

研究执行生命周期约定：

- 一次 `dispatch_product_search` / `dispatch_category_research` 对应一次临时 research run
- research run 内部只有一个子 Agent 上下文，由 MAF 管理多轮工具调用循环
- run 完成后销毁上下文，不长期保留子 Agent 的中间工具历史

**示例**：
- 用户在中国，用中文提问 → "以中文搜索为主，英文搜索为辅。中文搜索关键词应包含产品名称、评测、推荐等；英文搜索关键词用于补充国际评测源。"
- 用户在美国，用英文提问 → "仅使用英文搜索。搜索关键词应包含产品名称、review、best、buying guide 等。"
- 用户在中国，但询问国际品牌 → "优先看英文权威评测，再补中文用户经验。如果国际型号和国内型号不同，请标出差异。"

> [!IMPORTANT]
> 不将 location 作为搜索约束注入——避免干扰搜索广度和结果质量。

## 七、推荐策略规格

### 7.1 两轮推荐策略

**核心目的**：将用户放到场景中，通过用户对真实产品的反应来验证/校准之前的需求理解。两轮推荐是产品的**默认推进策略**，尤其适用于典型单品选购，但不应被理解为所有任务都必须被控制层强行压成固定两轮。

#### 第一轮：广度探索

- 推荐 **3-5 款**产品，覆盖用户需求的不同侧面（数量取决于候选质量，不强求 5 款）
- 不要求每个产品完美匹配所有需求，但整个集合要覆盖全部需求方向
- 每个产品配具体的推荐理由和说明
- 通过产品之间的差异，让用户自然感知到 tradeoff
- **直接主动请用户给出反馈/感受**

#### 第二轮：深度分析

根据用户对第一轮的反馈类型，决定第二轮的具体内容：

| 用户反馈类型 | 第二轮行动 |
|-------------|-----------|
| 觉得都不错，没有强方向 | 在已有候选上做**更深入的对比分析**，帮用户看清差异 |
| 有新偏好浮现 | 更新维度权重，在**已有候选中重新排序和分析** |
| 有明确新需求，当前候选不再适用 | 更新需求画像，**重新搜索** |
| 已有明确选择 | 提供该产品的**更多深度信息** |

> [!IMPORTANT]
> 第二轮不一定需要新搜索。可以在已有候选基础上做更深入的分析、排序、对比。只有用户提出明确新需求导致候选不适用时才重新搜索。

对于搭配/补齐/升级类任务，主 Agent 可以在默认两轮节奏内完成，也可以先给出阶段性建议、候选优先级或需要补充搜索的方向，再继续推进；关键是帮助用户收敛决策，而不是机械走完固定轮次。V1 中若当前焦点从一个子问题切换到另一个子问题，应将其视为围绕同一 `goal_summary` 的下一段顺序流程，而不是维护多个并行候选池。

### 7.2 反事实解释能力

在第二轮推荐中，每个推荐应能回答：

1. **为什么推荐这个？**（正面解释）
2. **为什么不是 XXX？**（用户可能期望的产品为什么没推荐）
3. **如果更在意 Y，结论会怎样变化？**（敏感性分析）

### 7.3 `recommendation_round` 状态管理

| 值 | 含义 |
|----|------|
| `"未开始"` | 尚未进入推荐阶段 |
| `"第一轮"` | 正在进行广度推荐 |
| `"第二轮"` | 正在进行深度分析 |
| `"完成"` | 推荐周期结束，触发画像更新 |

`recommendation_round` 是**描述性状态**，用于标记当前推荐节奏和触发后续处理；它不是控制层的严格状态机，不要求所有任务都机械经历完全相同的轮次。

**重置规则**：系统在处理 `dispatch_product_search` 后自动重置为 `"未开始"`（确定性系统行为）。

**完成规则**：主 Agent 在 session_updates 中将 `recommendation_round` 设为 `"完成"`（语义判断由 LLM 做），触发系统自动执行画像更新。

## 八、用户画像规格

### 8.1 宏观画像（Global Profile）

记录跨品类稳定的用户特征，用于在新品类或新会话中复用消费风格、决策习惯和生活方式背景。

**信息获取方式**：

| 方式 | 信息类型 | 时机 |
|------|---------|------|
| 轻量前置收集（onboarding） | 性别、年龄段、所在城市 | 首次使用时 |
| 对话推断 + 延迟应用 | 消费水平、职业等 | 推荐完成后生成草稿，并在下次启动恢复检查时应用 |
| 品类知识驱动按需获取 | 品类特有的用户属性（如体型） | 对话中由 urgency 机制驱动 |

**包含字段**：demographics（基础人口统计）、consumption_traits（消费行为特征）、lifestyle_tags（生活方式标签）、notes。

这些信息的目标不是替代当次需求挖掘，而是在新任务开始时减少重复确认，并为跨品类迁移提供低成本的初始判断依据。

### 8.2 品类偏好（Category Preferences）

只存**确实可跨产品类型复用**的信息：主要使用场景、品牌偏好、消费倾向、选购记录。

**关键字段**：
- `consumption_traits`：包含品类消费倾向、preferred_brands、**anti_preferences**（排斥偏好）
- `primary_scenarios`：主要使用场景
- `purchase_history`：选购记录
- `profile_source`：画像来源标记（is_native / migrated_from）

> [!IMPORTANT]
> **`anti_preferences`（排斥偏好）作为独立字段存在。** 负面信号（"绝对不要XX"）往往比正面信号更稳定、筛选价值更高。
>
> `preferred_brands[*].reason` 和 `anti_preferences[*].reason` 为**可选字段**。如果用户只表达了偏好/排斥对象，没有明确原因，可以只记录对象本身。

对于搭配/补齐/升级任务，只有那些在后续对话中有稳定复用价值的信息才应写回品类偏好，例如核心使用场景、稳定品牌倾向、最近一次选购记录。当前会话中的临时搭配细节不应被过度固化。

### 8.3 画像更新规则

> [!IMPORTANT]
> **这是相对前期产品文档的一项明确设计变更。**
>
> 前期文档将长期画像写入时机定义为“会话结束时批量更新”。当前 SPEC 改为“推荐完成时生成 `pending_profile_updates` 草稿，下次启动恢复检查后再正式应用”。这样做的原因是：当前 V1 是本地 CLI 形态，没有可靠的在线会话结束事件；如果继续依赖“会话结束”，用户直接退出会导致有效信号丢失。

| 更新时机 | 触发条件 | 更新范围 |
|---------|---------|---------|
| **推荐周期结束** | 系统检测 `recommendation_round = "完成"` | 将 `DecisionOutput.profile_updates` 保存为 session 内的 `pending_profile_updates` 草稿，不直接写入长期画像 |
| **下次启动恢复检查** | 发现活跃 session 中存在未应用的 `pending_profile_updates` | 系统二次检查 intent、异常状态和收敛情况后，写入长期画像 |
| **Onboarding** | `onboard_user` action | 仅写入 demographics |

**设计原则**：

- 长期画像不依赖“用户明确说我很满意”这类额外确认
- 也不依赖严格意义上的“会话结束事件”
- `recommendation_round = "完成"` 只表示已经形成了可提炼的画像草稿
- 真正写入长期画像，由系统在下次启动时基于保留的 session 做恢复检查后执行

**intent 敏感性规则**（系统层在应用 `pending_profile_updates` 时检查）：

| intent | 画像更新行为 |
|--------|-------------|
| 自用选购 | 标准流程，更新自用消费画像 |
| 送礼 | **不更新**自用消费画像（本次需求不代表用户自身偏好） |
| 复购/换代 | 标准流程 |
| 纯咨询 | **不更新**消费画像（探索性行为不代表真实偏好） |

**画像迁移机制**：在新品类中使用从其他品类迁移的偏好时，`is_native` 标记为 `false`，相关维度 confidence 统一标为 4（未确认）。

### 8.4 「越用越懂」体验路径

| 使用次数 | 系统掌握的信息 | 用户体验变化 |
|---------|--------------|------------|
| 第 1 次 | 全部从零开始 | 需要 5-8 轮需求挖掘 |
| 第 2 次（同品类不同产品类型） | 已有场景、品牌偏好、消费倾向 | 只需 2-3 轮确认产品类型特有需求 |
| 第 3 次（不同品类） | 已有消费特征、决策风格 | 跳过通用信息，直接进入品类需求 |
| 第 N 次 | 丰富的用户画像 + 多品类知识库 | 接近「老朋友帮你选」的体验 |

## 九、Intent 行为模式映射

Agent 从对话中自然识别用户意图类别，不需要显式询问。不同意图会触发不同的系统行为。未匹配到任何特定模式时，按「自用选购」默认流程执行。

| intent | 识别信号 | 需求来源 | 画像更新 | 决策压力 |
|--------|---------|---------|---------|---------|
| **自用选购**（默认） | 无特殊信号 | 用户自身需求 | 标准流程 | 标准 |
| **送礼** | 「送人」「礼物」「给朋友/家人买」 | 收礼人需求（需额外了解收礼人信息） | 不更新自用消费画像 | 标准 |
| **复购/换代** | 「换掉」「升级」「用了几年想换」 | 复用已有产品使用体验作为需求基准 | 标准流程 | 标准 |
| **纯咨询** | 「了解一下」「还没想好要不要买」「先看看」 | 用户自身需求（但可能不明确） | 不更新消费画像 | 低——减少推进压力，侧重信息提供 |

## 十、错误处理与降级规格

### 10.1 约束冲突检测

**执行者**：主 Agent 在 CoT 中完成语义级约束检查。系统层仅做 payload 完整性和基础合法性的 sanity check（如必填字段、空值、基础类型），**不做 tradeoff/预算覆盖/可调和性判断**。

| dispatch 类型 | 是否需要约束检查 | 原因 |
|--------------|----------------|------|
| 品类调研 | **否** | 此时尚无品类知识，无法检查 |
| 产品搜索 | **是** | 已有品类知识和用户需求 |
| 用户反馈后的重新搜索 | **轻量检查** | 仅验证变化维度是否产生新冲突 |

**约束冲突类型**：

| 类型 | 描述 | 处理方式 |
|------|------|---------|
| `budget_vs_requirement` | 预算远低于满足需求的最低价位 | 解释价位-性能关系，提供双向替代方案 |
| `tradeoff_conflict` | 两个 priority=4 的维度之间存在 tradeoff | 用 tradeoff 知识教育用户，引导取舍 |
| `no_matching_product` | 需求组合过于独特，搜索返回极少结果 | 解释原因，建议放宽最不关键的约束 |

**沟通原则**：不否定用户，提供选择。先解释为什么冲突（教育性）→ 给出双向替代方案 → 让用户自己选择方向。

### 10.2 研究 Agent 错误处理

| 错误类型 | 处理层 | 行为 |
|---------|-------|------|
| 搜索工具失败 | 应用层 + 研究 run | 应用层可重试 1 次；重试时补充 retry `research_brief` 提示模型更换关键词与扩大来源覆盖；仍失败则降级返回可消费的失败结果 |
| 结果不足 / 覆盖面有限 | 研究 Agent 同次 run 内部 | 优先继续搜索、换 query、补充来源；不定义固定的产品数阈值或固定的扩搜算法；若仍不足，返回已有结果或空列表并说明限制 |
| 来源可靠性 | 研究 Agent | 标注 `source_consistency` 字段，在 `notes` 中标注仅有单一来源的数据 |
| 语义校验失败 | 主 Agent（CoT） | 检查产品是否匹配硬约束、描述是否自洽、是否存在幻觉迹象 |

**主 Agent 语义校验规则**：
- 产品是否真正匹配用户硬约束（price 在预算范围内？场景匹配？）
- 若 `products` 非空，则每个产品都应具备 `name / brand / price / specs / features / pros / cons / sources / source_consistency` 必填字段；若为空列表，应视为“没有匹配结果”的合法返回，并结合 notes 向用户解释原因
- `search_meta` 必须提供结构化运行状态，至少包含重试次数、结果状态、是否扩大搜索范围；应用层据此稳定写入 `error_state.search_retries` 与相关 `events`
- `search_meta.retry_count` 和 `search_meta.search_expanded` 由应用层视运行过程做规范化，不能简单假设为模型自报 telemetry
- price 在合理范围内（非 $0 或异常高价）
- 产品描述是否自洽（features/pros/cons 不矛盾）

**幻觉检测信号**：`source_consistency: low`、`sources` 字段为空或无效 URL、产品名称/品牌组合不常见。检测到时降低该产品的推荐优先级，推荐时标注「信息待验证」。

### 10.3 推荐阶段异常处理

| 异常 | 阈值 | 处理 |
|------|------|------|
| 连续负面反馈 | ≥ 2 轮 | Agent 在 CoT 中反思；主动请用户说明不满意的具体原因，重新锚定核心需求 |
| 需求反复漂移 | 维度权重修改超过 3 次 | 主动做需求总结确认；持续发生则切换到纯咨询模式 |

### 10.4 降级优先级总原则

```
能力完整 ──→ 局部降级 ──→ 告知限制 ──→ 建议离开
```

1. 优先保证用户始终能得到某种程度的帮助（不要返回空结果 + 什么都不说）
2. 降级时明确告知用户发生了什么、数据可靠性如何（透明度）
3. 任何时候都提供下一步行动建议（不要让用户无所适从）
4. Agent 的限制是可以承认的——比假装能做更好

## 十一、Session 管理规格

### 11.1 Session State

会话状态文档统一包含决策推进状态 + 需求画像快照，「我知道了什么」和「我该做什么」互相校验。

**核心顶层字段**：`session_id`、`last_updated`、`intent`、`product_type`、`category`、`decision_progress`、`requirement_profile`、`jtbd_signals`、`error_state`。

当任务属于搭配/补齐/升级时，session 可轻量承载与当前目标直接相关的辅助信息，例如 `goal_summary`、`existing_items`、`missing_items`。这些字段是可选的，只在当前任务需要时出现。

`error_state` 在 SPEC 中定义为“固定核心字段 + 可扩展事件”的结构：

```json
{
  "error_state": {
    "constraint_conflicts": [],
    "search_retries": 0,
    "consecutive_negative_feedback": 0,
    "validation_warnings": [],
    "events": []
  }
}
```

其中：

- `constraint_conflicts`、`search_retries`、`consecutive_negative_feedback`、`validation_warnings` 为固定核心字段，便于统计和验收
- `events` 用于记录额外异常事件，例如 `insufficient_results`、`partial_search_result`、`search_failed`、`stale_session_recovery` 等；这些事件默认由应用层根据研究结果中的结构化 `search_meta` 与运行上下文机械生成，而不是从自由文本 `notes` 反推

除核心顶层字段外，session 还允许出现以下**可选运行时顶层字段**：

- `pending_research_result`：研究 Agent → 主 Agent 的一次性交接包装（轻量结构：`{ "type": "category_research" | "product_search", "result": {...} }`）。系统在主 Agent 完成下一轮消费后清除，不依赖 `session_updates` 是否存在（见 §11.4）。
- `pending_profile_updates`：长期画像更新草稿。当 `recommendation_round = "完成"` 时由系统写入 session，在下次启动恢复检查时决定是否正式应用到 `global_profile` 和 `category_preferences`。
- `candidate_products`：当前推荐周期的候选池，承载研究 Agent 初筛得到的活动候选集合，供第一轮和第二轮推荐复用。对于搭配/补齐类任务，V1 中它始终只对应当前焦点的那一批候选；切换焦点时允许刷新，但不承担多子问题并行保留职责。

> 完整 Schema 定义见 INTERFACES.md。

### 11.2 `decision_progress.stage`

stage 是**标签而非严格状态机**，允许非线性跳转。

可选值及与六阶段流程（§3.1）的对应关系：

| stage 值 | 对应流程阶段 |
|---------|------------|
| `"品类认知构建"` | 阶段 1 |
| `"需求挖掘"` | 阶段 2 |
| `"产品搜索"` | 阶段 3（研究 Agent 执行搜索期间） |
| `"候选探索"` | 阶段 4（两轮推荐期间） |
| `"会话收尾"` | 阶段 5 |

### 11.3 Session 生命周期

| 场景 | 系统行为 |
|------|---------|
| 新会话 | 创建 `current_session.json`，`session_id` 格式为 `{date}-{HHMMSS}`（如 `2026-04-10-143052`） |
| 下次启动发现已有活跃 session | 先执行恢复检查：处理未应用的 `pending_profile_updates`，再决定继续该 session 或基于它开启新对话 |
| 会话暂停超过 7 天 | 在 context 中注入 staleness 标注，由主 Agent 判断如何恢复 |
| 历史保留 | 不主动清空或删除 session 文件；历史 session 可长期保留，但默认不进入主 Agent context |

### 11.4 `session_updates` 写入规则

- 采用 **section 级替换**：Agent 输出更新后的完整 section，系统做 key 覆盖
- 合法的顶层 key 采用**白名单机制**；该机制作用于 `DecisionOutput.session_updates`，白名单内的 key 允许新增和覆盖。完整 session state 的持久化不以此白名单做过滤
- `pending_research_result` 是一次性交接字段：系统在主 Agent 完成下一轮消费后清除，不依赖 `session_updates`
- `candidate_products` 是推荐周期内的活动候选池：在 `dispatch_product_search` 后刷新，在推荐阶段跨两轮复用，新的产品搜索会覆盖旧候选池。对搭配/补齐/升级类任务，这种覆盖行为是 V1 的预期设计，表示当前焦点顺序切换
- `pending_profile_updates` 由系统在 `recommendation_round = "完成"` 时写入，在下次启动恢复检查时应用或丢弃
- 默认 context 只加载当前活跃 session；历史 session 保留但不默认读取

## 十三、非功能性需求

### 13.1 边界保护

| 保护机制 | 阈值 | 触发行为 |
|---------|------|---------|
| 单 session 最大对话轮数 | 30 轮 | 提示用户保存并结束 |
| 需求挖掘阶段最大追问轮数 | 8 轮 | Agent 必须推进或说明困难 |
| `dispatch_category_research` 单 session 建议次数 | 2 个不同品类 | 当已调研 2 个不同 category、准备进入第 3 个不同品类前触发软提示，要求主 Agent 优先复用已有 knowledge，并解释继续调研的必要性 |
| `dispatch_product_search` 单 session 建议次数 | 6 次 | 超过后触发软提示，要求主 Agent 优先复用已有候选或解释限制 |
| 连续负面反馈检测 | ≥ 2 轮 | 触发需求重新锚定 |

### 13.2 Context 管理

| 组成部分 | 预估大小 |
|---------|---------|
| System Prompt | 3-5k tokens |
| 品类知识（选择性加载后） | 2-4k tokens |
| 用户画像 | 0.5-1k tokens |
| Session state | 1-2k tokens |
| 研究结果（如有） | 2-3k tokens |
| 对话历史 | 5-15k tokens |
| **总计** | **~15-30k tokens** |

现代模型 context window（128k-200k+）远大于此。V1 不引入 context 压缩或 reset 机制。

### 13.3 System Prompt 策略优先级

System Prompt 在产品层面应遵循以下策略优先级层次，确保运行时行为顺序一致：

1. **前置检查**
   - `dispatch_product_search` 前执行约束冲突检测
   - 收到研究结果后执行语义级校验
2. **每轮必做**
   - 分析用户输入，批量更新相关维度的 `priority` 和 `confidence`
   - 检查 urgency 阈值，决定继续追问、进入搜索或进入推荐分析
3. **触发式策略**
   - 遇到模糊表达时触发锚定确认
   - 用户反馈候选时触发偏好漂移检测
   - 搭配/补齐/升级任务中，必要时触发已有物品确认
4. **阶段性策略**
   - 品类知识缺失时触发品类调研
   - 进入推荐阶段后按默认两轮策略推进
   - `recommendation_round = "完成"` 时生成 `pending_profile_updates`
   - 下次启动发现活跃 session 时执行恢复检查并决定是否应用长期画像更新

更具体的 prompt 设计细节见 `PROMPTS.md`，运行时资源文件位于 `src/prompts/`；运行顺序应以本节为准。

### 13.4 模型与运行时配置

| 变量 | 说明 |
|------|------|
| `LLM_BASE_URL` | 共享默认 LLM endpoint |
| `LLM_API_KEY` | 共享默认 LLM key |
| `MAIN_AGENT_MODEL` | 主 Agent 模型名 |
| `RESEARCH_AGENT_MODEL` | 研究 Agent 模型名 |
| `MAIN_AGENT_BASE_URL` | 主 Agent 专属 endpoint（可选） |
| `MAIN_AGENT_API_KEY` | 主 Agent 专属 key（可选） |
| `RESEARCH_AGENT_BASE_URL` | 研究 Agent 专属 endpoint（可选） |
| `RESEARCH_AGENT_API_KEY` | 研究 Agent 专属 key（可选） |
| `TAVILY_API_KEY` | Tavily 搜索 key |

优先级约定：

- Agent 专属配置优先于共享默认配置
- 共享默认配置优先于代码默认值

> [!WARNING]
> 研究 Agent 可以使用成本较低但能力仍较强的模型，以换取更大的搜索广度；但主 Agent **必须对研究 Agent 的输出做语义级校验**，不能把研究结果直接视为完全可靠事实。

## 十四、验收标准

### 14.1 核心评估指标

#### 指标 1：需求缺失率（Requirement Miss Rate）

| 项目 | 定义 |
|------|------|
| **含义** | 在推荐阶段才被发现遗漏的关键需求维度占比 |
| **计算** | 遗漏维度数 / 用户最终确认的总关键维度数 |
| **数据来源** | session 日志：在阶段 4 中用户提出的新需求且 priority ≥ 3 的维度 |
| **目标** | < 15% |

#### 指标 2：阈值拟合度（Threshold Calibration）

| 项目 | 定义 |
|------|------|
| **含义** | urgency 阈值判断（进入搜索/推荐的决策点）的准确度 |
| **计算** | 阈值判断正确的次数 / 总阈值判断次数。「正确」指进入搜索后没有因信息不足回退到需求挖掘 |
| **数据来源** | session 阶段流转日志：统计阶段 3→2 和阶段 4→2 的回退次数 |
| **目标** | 回退率 < 20% |

#### 指标 3：推荐采纳率（Recommendation Acceptance Rate）

| 项目 | 定义 |
|------|------|
| **含义** | 用户在推荐列表中表示正面兴趣的产品比例 |
| **计算** | 用户表示感兴趣的产品数 / 推荐产品总数 |
| **数据来源** | 分析用户对第一轮推荐的反馈（自然语言情感分析） |
| **目标** | 第一轮 ≥ 40%（5 个里至少 2 个感兴趣） |

#### 指标 4：研究 Agent 搜索效率

| 项目 | 定义 |
|------|------|
| **含义** | 研究 Agent 完成一次搜索任务的资源消耗 |
| **计算** | 平均搜索耗时、平均 API 调用次数、waste_rate = (搜索返回产品数 - 最终推荐使用数) / 搜索返回产品数。「最终推荐使用数」指在任意一轮推荐中被主 Agent 引用的不重复产品数 |
| **数据来源** | 研究 Agent 执行日志 |
| **目标** | waste_rate < 40%（7 个返回至少用 4 个） |

### 14.2 端到端场景验收

以下场景必须完整跑通。以冲锋衣选购为参考示例，展示每个阶段的预期行为模式。

---

#### 场景 A：完整 Happy Path（新用户 × 新品类 × 全流程）

**阶段 0：初始化 + Onboarding**
```
[系统] ProfileContextProvider 检测到 global_profile.json 不存在（新用户）
       → context 注入 onboarding 标注

主 Agent → 输出 ask_user，提问性别/年龄段/城市
用户 → "男，28岁，在上海"
主 Agent → 输出 onboard_user，携带 demographics payload
[系统] → 写入 global_profile.json

验证点:
  ✓ ProfileContextProvider 正确检测新用户
  ✓ onboard_user action 的 payload 包含 gender/age_range/location
  ✓ global_profile.json 被正确创建
```

**阶段 0 续 → 阶段 1：品类认知构建**
```
用户 → "想买一件冲锋衣"
主 Agent → 判断 category="户外装备", product_type="冲锋衣"
         → 检查 knowledge/户外装备.json → 不存在
         → 输出 dispatch_category_research

[系统] → 创建研究 Agent → 搜索品类知识 → 返回结构化 CategoryResearchOutput
[系统] → 写入 knowledge/户外装备.json（category_knowledge + product_types.冲锋衣）
       → 包装为 pending_research_result 放入 session

主 Agent → 收到品类知识
         → 初始化需求画像（所有维度 priority 待定，confidence=4）
         → session stage → "需求挖掘"

验证点:
  ✓ 正确识别品类和产品类型
  ✓ 品类文档不存在时触发 dispatch_category_research
  ✓ 知识文档包含 category_knowledge + product_types.冲锋衣 两层
  ✓ pending_research_result 被正确设置和清除
```

**阶段 2：需求挖掘（2 轮对话）**
```
第 1 轮: 批量提问
  主 Agent [评估阻塞维度]:
    使用场景: priority=4, confidence=4, urgency=16 → 阻塞
    预算:     priority=4, confidence=4, urgency=16 → 阻塞
    性别:     priority=4, confidence=1, urgency=4  → 已解决（onboarding）
    → 存在多个 urgency≥12 维度，批量提问

  主 Agent → "你主要在什么场景穿？预算大概多少？有没有品牌偏好？"

  用户 → "周末经常去爬山，偶尔高海拔。预算3000左右。品牌没偏好，
         之前淋过雨普通冲锋衣不太行，防水要好些。"

  主 Agent [一段话覆盖多个维度]:
    使用场景: confidence=2（"高海拔"模糊 → 触发 ambiguity_risk）
    预算: value="3000左右", confidence=1
    防水: priority=4, confidence=1（用户直接表达）
    JTBD 信号: 情绪（不想重蹈覆辙）

第 2 轮: 歧义锚定
  → "高海拔"仍模糊 → 唯一需要确认的点
  主 Agent → "你说的高海拔是3000-4000米景区步道，还是5000米以上冰雪攀登？"
  用户 → "四姑娘山那种，最高4000多米"
  主 Agent → 使用场景 confidence=1（锚定确认）

  阈值检查:
    所有维度最高 urgency=9（透气/耐久，推断值）
    无≥12 阻塞，8-11 区间维度=2 个（≤3）
    → 进入产品搜索

验证点:
  ✓ 批量提问而非逐维度轮询
  ✓ 一段用户回答正确更新多个维度
  ✓ ambiguity_risk 触发锚定确认（"高海拔"→具体化）
  ✓ 阈值判断正确——2 轮后无阻塞 → 推进
  ✓ 不追问非阻塞的推断值（透气/耐久 urgency=9）
```

**阶段 3：产品搜索**
```
主 Agent → 输出 dispatch_product_search，payload 包含:
  product_type: "冲锋衣"
  search_goal: "适合周末徒步和4000m级高海拔的冲锋衣"
  constraints: { budget: "2500-3500", gender: "男款",
                 key_requirements: ["高防水"], scenario: "周末徒步+4000m" }

研究 Agent → 执行搜索（中文+英文）→ 返回 7 款候选（结构化 ProductSearchOutput）
  → 每款包含 name/brand/price/specs/features/pros/cons/sources/source_consistency

[系统] → 包装为 pending_research_result
       → 刷新 candidate_products（保存 research agent 初筛候选池）
       → recommendation_round 重置为 "未开始"

验证点:
  ✓ dispatch payload 包含 exclusions 字段（即使为空）
  ✓ 研究 Agent 输出包含完整的 features 介绍
  ✓ source_consistency 字段已标注
  ✓ candidate_products 保存的是研究 Agent 初筛候选池，而不是主 Agent 第一轮展示的子集
  ✓ recommendation_round 被系统自动重置
```

**阶段 4：两轮推荐**
```
第一轮（广度探索）:
  主 Agent → 从 candidate_products 的 7 款中选出 5 款，覆盖不同侧面:
    综合型 / 耐用性价比 / 极致轻量 / 透气优先 / 预算友好
    → 每款附推荐理由 + 注意事项
    → 主动请用户反馈

  用户 → "国产那个不考虑。1和3感兴趣。在意耐不耐用。"

  主 Agent [多信号分析]:
    → anti_preference: "国产品牌" → 记录
    → JTBD: 品牌信任=安全感
    → 耐久性: priority 3→4, confidence=1（用户直接说）
    → 无 urgency≥12 → 不回退，基于已有候选分析

第二轮（深度分析）:
  主 Agent → 基于同一批 candidate_products 对比关键矛盾（轻量 vs 耐久 tradeoff）
    → 用面料 Denier 数据量化差异
    → 提供条件化建议（"如果更在意 X → 选 A"）
    → 反事实解释（"Montbell 虽轻但和你的耐久需求冲突"）

  用户 → 做出选择

验证点:
  ✓ 第一轮 3-5 款，覆盖不同需求方向
  ✓ 用户反馈正确触发维度更新（耐久 priority 提升）
  ✓ anti_preferences 被记录
  ✓ 第二轮基于 candidate_products 深度分析（未重新搜索）
  ✓ 提供了反事实解释
```

**阶段 5：会话收尾**
```
主 Agent → recommendation_round = "完成"（通过 session_updates）

[系统] 检测到 "完成" → 将 profile_updates 写入 session.pending_profile_updates:
  → 暂不直接更新 global_profile.json / category_preferences
  → 暂不清空 current_session.json

[下次启动]
  系统检测到存在未应用的 pending_profile_updates
  → 执行恢复检查（intent / error_state / 收敛情况）
  → 如通过检查，再更新:
    global_profile.json:
      consumption_traits（决策风格、品牌态度）
    category_preferences/户外装备.json:
      品类消费倾向 + preferred_brands + anti_preferences
      primary_scenarios + purchase_history

验证点:
  ✓ recommendation_round="完成" 生成 pending_profile_updates 草稿
  ✓ 当前 session 被保留，未被强制清空
  ✓ 下次启动时执行恢复检查
  ✓ intent 检查执行（自用选购 → 标准更新）
  ✓ global_profile 和 category_preferences 在恢复检查后被更新
  ✓ purchase_history 中记录了本次选购
```

#### 场景 B：回退场景（推荐阶段发现新需求 → 重新搜索）

```
[接场景 A 第一轮推荐后]

用户 → "这几个都不太满意，我想要看起来很专业的那种"

主 Agent [分析]:
  - "看起来很专业" → 新的需求维度（外观/专业感），之前 priority=1
  - 偏好漂移（显式）
  - JTBD: 社会信号（想通过装备表达身份）
  - "看起来专业" → ambiguity_risk 高 → 触发锚定确认

主 Agent → "你说的'看起来专业'是指：
           低调技术感（简洁设计），还是功能细节丰富（多口袋、挂点）？"

用户 → "功能细节多的那种，一看就是玩户外的"

主 Agent → 更新: 外观/专业感 priority=3, confidence=1
         → 评估: 已有候选不含"登山款外观"需求 → 需重新搜索
         → 输出 dispatch_product_search（新 search_goal 包含外观需求）

[系统] → recommendation_round 自动重置为 "未开始"
       → 研究 Agent 搜索 → 返回新候选
       → 重新进入第一轮推荐

验证点:
  ✓ 新需求维度正确添加到 session（priority/confidence）
  ✓ 歧义触发锚定确认
  ✓ 正确判断需要重新搜索（而非在已有候选中分析）
  ✓ recommendation_round 被系统自动重置
  ✓ 新搜索条件包含更新后的需求
```

#### 场景 D：错误恢复场景

```
主 Agent → 输出 dispatch_product_search

研究 Agent [搜索失败]:
  → 搜索工具第一次失败 → 自动重试（换关键词）
  → 在同一次 research run 内继续搜索、主动补充来源覆盖
  → 结果仍偏少 → 返回已有结果 + notes 说明搜索限制

主 Agent [收到不完整结果]:
  → CoT 语义校验: 2 款产品字段完整，但覆盖面不足

  → 向用户透明说明: "这个价位段符合条件的产品比较少，

  → session error_state.events 记录: [{ type: "insufficient_results", details: ... }]

验证点:
  ✓ 研究 Agent 自动重试并尝试继续扩展搜索
  ✓ 不足结果不导致崩溃，返回已有结果 + 说明
  ✓ 主 Agent 对用户透明沟通限制
  ✓ error_state 正确记录
  ✓ 提供下一步建议（非空白终止）
```

## 十五、词汇表

| 术语 | 含义 |
|------|------|
| **priority**（权重） | 1-4 量表，衡量决策维度对用户的重要程度。4=关键/硬约束 |
| **confidence**（置信度） | 1-4 量表，衡量 Agent 对某信息的确定程度。1=确定，4=未知 |
| **urgency**（紧迫度） | `priority × confidence`，分数越高越需要优先处理 |
| **stage** | Session 中的 `decision_progress.stage`，标记当前所处业务阶段。是标签而非严格状态机 |
| **intent** | 用户意图类别（自用选购/送礼/复购/纯咨询），影响画像更新和决策压力 |
| **category**（大品类） | 品类的最高级别分类（如「户外装备」「数码电子」），knowledge 和 category_preferences 按此维度组织 |
| **product_type**（产品类型） | 大品类下的具体产品类型（如户外装备下的「冲锋衣」「登山鞋」） |
| **锚定确认** | 遇到用户的模糊/抽象表述时，用具体数字或场景来确认用户真实需求的策略 |
| **dispatch** | 主 Agent 通过 `next_action` 指示系统层创建研究 Agent 执行搜索任务 |
| **onboarding** | 首次用户的轻量信息收集流程（性别/年龄段/城市） |
| **JTBD**（Jobs to Be Done） | 需求分析框架，将用户需求分为功能任务、情绪任务、社会任务三层 |
| **tradeoff** | 决策维度之间的此消彼长关系（如轻量 vs 耐久） |
| **recommendation_round** | 标记当前推荐周期的进度（未开始/第一轮/第二轮/完成） |
| **anti_preferences** | 用户的排斥偏好（如排斥特定品牌），负面信号往往比正面信号更稳定 |
| **DecisionOutput** | 主 Agent 每轮输出的 Pydantic 结构化模型，包含用户消息、推理过程、下一步动作、session 更新等 |
| **session_updates** | DecisionOutput 中声明的 session state 变更，采用 section 级替换 |
| **profile_updates** | DecisionOutput 中声明的画像变更数据，仅在 `recommendation_round = "完成"` 时填充 |
| **pending_research_result** | Session 中的一次性交接字段，存放研究 Agent 的最新输出，供主 Agent 下一轮消费后由系统清除 |
| **candidate_products** | Session 中的活动候选池，保存研究 Agent 初筛后的产品列表，供第一轮和第二轮推荐复用 |
| **source_consistency** | 研究 Agent 输出中标记产品信息来源一致性的字段（high/medium/low） |
